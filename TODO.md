## 🔴 最值得做(性价比高)

- [x] **列表图缩略图(thumbnail)** ✅
  懒生成 + 磁盘缓存的 `/thumb/<path>` 端点(600px WebP),grid / 检查点列表用小图、放大与对比仍用原图;
  随批次删除/覆盖级联清理、`prune_orphans` 清孤儿、`@error` 回退原图。
  效果:单个 grid 的图片流量从 ~100MB 降到 ~15MB(≈6.7×),缩略图响应头带 `max-age` 浏览器缓存。
  **仍存在的尾延迟**见下条(请求**数量**问题,非字节问题)。

- [ ] **列表图懒加载 / 虚拟化(削减请求数)**
  缩略图把每张变小了,但 grid 仍**一次性加载全部单元格**(实测 32×12≈383 个请求);多人同时打开时尾延迟主要来自请求数 + 单进程后端吞吐,而非字节。
  做法:只渲染/加载**视口内**的单元格(IntersectionObserver 或网格虚拟化),离屏不请求。**这是 grid 并发体验的下一个主要杠杆。**

- [x] **`/images` 静态图加缓存头** ✅
  `StaticFiles` 子类给原图加 `Cache-Control: public, max-age=86400`(仍保留 ETag/Last-Modified 供条件请求)。
  放大/详情二次查看直接命中浏览器缓存,不重复下载。

- [x] **批次删除的前端入口** ✅
  批次列表操作列末尾加低调的小垃圾桶(默认浅灰、悬停变红)+ `a-popconfirm` 二次确认(含级联说明);
  走 `store.deleteBatch` → `DELETE /api/batches/{id}`,删后清掉本地所选/按场景记忆并刷新。
