# PixelComparison 使用文档

游戏截图视觉回归对比平台:采集模块上报各版本的场景截图,平台对任意两个批次做像素级对比,
输出差异率、SSIM、PSNR 等指标和差异热力图,帮助 QA 快速定位渲染回归。

---

## 1. 系统架构

```
游戏端 / CI 采集模块
      │  POST /api/batches + /screenshots(上报批次与截图)
      ▼
FastAPI 后端 ──── SQLite(批次/截图/基线/对比结果)
      │   对比引擎:Pillow + numpy(差异率 / SSIM / PSNR / 热力图 / 直方图)
      │   静态服务:/images(截图与热力图文件)
      ▼
Vue 3 + Arco Design 前端(批次管理 / 对比查看)
```

| 目录 | 说明 |
|---|---|
| `backend/app/compare.py` | 对比引擎:像素 diff、SSIM、PSNR、热力图、直方图 |
| `backend/app/service.py` | 对比服务(按场景名配对)、基线晋升、状态阈值 |
| `backend/app/models.py` | Batch / Screenshot / Baseline / Comparison / ComparisonItem |
| `backend/app/main.py` | REST API + 静态图片服务 |
| `backend/app/seed.py` | 演示数据生成(可重复执行,会清库重建) |
| `frontend/src/store.js` | 前端全部状态与数据加载(Pinia) |
| `frontend/src/components/` | 页面组件 |

## 2. 安装与启动

### 环境要求

- Python ≥ 3.11
- Node.js ≥ 18

### 后端

```powershell
cd backend
python -m venv .venv
.venv\Scripts\pip install -r requirements.txt
.venv\Scripts\python -m app.seed        # 可选:生成演示数据(清库重建)
.venv\Scripts\python -m uvicorn app.main:app --port 8000 --reload
```

- 交互式 API 文档:http://127.0.0.1:8000/docs
- 数据与图片存放在 `backend/data/`(SQLite 库 + 图片目录),删除该目录即彻底重置

### 前端

```powershell
cd frontend
npm install
npm run dev          # 开发:http://localhost:5173(已配置 /api 与 /images 代理)
npm run build        # 生产构建,产物在 dist/
```

### 一键开发启动(双控制台 + 日志)

```powershell
.\run-dev.ps1        # 开两个控制台:后端(8000,--reload) + 前端(5173),实时显示日志
```

日志位置 `backend/data/logs/`(滚动文件,5MB×5):
- `backend.log` — 后端请求(`→/←` + 耗时)与关键业务事件(建/删批次、发起对比、对比完成/失败、自动对比),中文。
- `frontend.log` — 前端 console / 报错(经 `POST /api/client-logs` 上报落盘)。

## 3. 核心概念

| 概念 | 含义 |
|---|---|
| **批次 Batch** | 一次截图采集运行,带**场景ID**(`scene_id`,UE Level)、**P4版本**(`p4_version`,可空)、平台、画质(`shading_quality`);由采集模块或网页「手动上报」录入 |
| **截图/检查点 Screenshot** | 批次内某个检查点的一张图,以检查点名(`scene_name`)为配对键 |
| **基线 Baseline** | 被"晋升"为参照标准的批次,带版本号(如 v1.1.5),按项目 + 平台隔离;同版本旧基线自动退役 |
| **对比 Comparison** | 当前批次 × 参照批次的一次对比;参照批次任选,同一批次可与多个参照各比一次 |
| **对比项 ComparisonItem** | 单场景的对比结果 |

### 场景状态判定

按场景名配对两个批次的截图后,每个场景进入以下状态之一:

| 状态 | 条件 | 含义 |
|---|---|---|
| 通过 pass | 差异率 < 0.3% | 视为无变化 |
| 警告 warn | 0.3% ≤ 差异率 < 2% | 轻微差异,建议人工过目 |
| 失败 fail | 差异率 ≥ 2% | 显著差异,疑似渲染回归 |
| 新增 added | 仅当前批次有此场景 | 新场景,确认后可入基线 |
| 缺失 missing | 仅参照批次有此场景 | 当前批次未产出截图,按失败级处理 |

整个对比的状态:任一检查点 fail/missing → 失败;任一 warn/added → 警告;否则通过。

> 阈值在前端「项目设置」页可改(差异率红/橙阈值、像素差异阈值、热力图模糊/灵敏度),
> 持久化在数据库;默认值见 `backend/app/settings.py` 的 `DEFAULT_SETTINGS`。对新发起的对比生效。

## 4. 界面操作指南

页面有独立 URL:`/batches`(批次管理)、`/comparison`(对比结果)、`/settings`(项目设置);
`/batches/<场景ID>` 可直达该场景的「列表图」。顶栏右侧图标:**刷新**、**手动上报**、切换主题。

### 4.1 批次管理页(`/batches`)

- **顶部横向筛选条**:场景ID(可输入文本搜索)/ 画质 / 创建时间;选择即时生效,「清空」恢复默认。
  - **创建时间**两种模式:**范围**(默认近七天,不可清空成全部)、**指定日期**(逐个加入多天、跳着选,标签可单独删除)。
- **两种视图**(右上角「列表 / 列表图」切换):
  - **列表** — 批次表格(批次ID、场景ID、画质、P4、检查点数、创建时间)。每行可「预览」(画廊)、设为「基线批次 / 对比批次」;两者同场景ID时点顶部「发起对比」(异步执行、进度轮询,完成跳到对比结果页)。每页条数按可用高度自适应。
  - **列表图** — 需先选场景:把该场景所有批次排成图片矩阵(列=批次按创建时间**新→旧**、表头可临时折叠某列,行=检查点)。
    - **选择对比**:在列头直接点「基线 / 对比」(再点一次取消),**所选列按角色着色高亮**;该选择**按场景记忆**,切到别的场景再切回会自动恢复(同折叠状态一样,硬刷新才重置)。本视图不再单独显示顶部选择条。
    - **差异对比列**(最右、吸附):选好基线/对比后,若该对**已有对比缓存**则就地展示各检查点的差异热力图(只读命中,不触发计算);**无缓存**时表头的「差异对比」会变成「发起对比」按钮,点击就地计算、显示进度、**不跳转**,算完表头复原并填充热力图。表头以 VS 卡片展示两侧批次的日期与 P4(按角色着色)。
    - **浏览**:鼠标在矩阵上**按住拖拽**可上下左右平移;**单击图片放大**,放大后 `← →` 在同一检查点的各批次间翻看、`↑ ↓` 在同一批次的各检查点间翻看(到边界不循环)。
- **手动上报**(顶栏图标):弹窗中拖入一个 `PixelComparison` 数据包文件夹(含 `manifest.json` + `Screenshot/`),前端解析后按与脚本一致的流程上报(失败自动重试 3 次);批次号重复会提示;数据包缺 P4 时可手动填写。
- 删除批次目前通过接口 `DELETE /api/batches/{id}`(级联清理),前端未放按钮。

### 对比结果页(`/comparison`)顶部

- **对比对**:`基线 #x  VS  对比批次 #y`(各带画质),点击下拉切换历史对比;历史**最多保留 100 条**,新建超限自动淘汰最旧。
- **交换基线/对比**:一键把基线↔对比互换,两个方向都秒出结果;正反向**只占一条对比历史**。
- 右侧:场景ID、检查点数、整体差异率,以及「重新对比」(强制重算当前对比对)。
- 顶栏「刷新」在此页刷新对比历史。

### 4.2 检查点结果列表(左侧)

- 状态 chip(全部 / 失败 / 警告 / 通过,出现时还有 新增 / 缺失)点击过滤,数字为各状态计数
- 搜索框按检查点名模糊过滤(300ms 防抖)
- 每行:缩略图、检查点名、状态、差异率;点击在右侧查看详情
- 底部分页,默认每页 10 条

### 4.3 对比详情(中部)

- 标题行:场景名、状态、「上一个 / 下一个」(也可用键盘 ← →,顺序为场景名排序)
- 三种视图模式:
  - **三视图**:当前版本 / 参照版本 / 差异热力图并排(热力图越红差异越大)
  - **滑动对比**:两图叠放,拖动分割线或点击任意位置对比(单边缺图时不可用)
  - **原图对比**:只看两张原图
- 缩放:50% – 200%,超出容器可滚动
- 新增 / 缺失场景会显示提示条,缺图侧显示占位

### 4.4 指标面板(右侧)

| 指标 | 含义 |
|---|---|
| 差异率(像素) | 差异像素占总像素的百分比(核心判定指标) |
| 差异像素数 / 总像素数 | 绝对数量 |
| 最大差异(单像素) | 所有像素中最大的通道差值(0–255) |
| 平均差异(仅差异区域) | 只统计差异像素的平均差值 |
| RMS | 均方根误差,整体差异强度 |
| SSIM | 结构相似度,1 为完全一致(演示为全局简化版) |
| PSNR | 峰值信噪比(dB),越高越接近;99 表示完全一致 |
| 通道差异 | R/G/B 各通道的差异像素占比 |
| 直方图对比 | RGB 分布,实线填充 = 当前版本,虚线 = 参照版本 |

## 5. 采集模块接入(批次上报)

三种方式:① 网页「手动上报」拖入数据包文件夹;② 脚本 [`report.py`](../report.py)(调用详解见 [report脚本使用.md](report脚本使用.md));③ 直接调 API。
流程都是:建批次 → 逐张传图 →(可选)自动对比。

**脚本(最省事):**
```powershell
python report.py "C:\path\to\PixelComparison"          # 指向含 manifest.json 的数据包目录
python report.py ./pkg --host 10.30.129.32 --port 8000  # 指定后端地址
```

**直接调 API:**
```bash
# 1. 建批次(scene_id/platform 必填;id 省略则按已有数字号自增;p4_version/shading_quality 可省略)
curl -X POST http://127.0.0.1:8000/api/batches \
  -H "Content-Type: application/json" \
  -d '{"scene_id":"Lv_Starfall","platform":"WindowsEditor","p4_version":251200,"shading_quality":4}'

# 2. 上传截图(每个检查点一张,scene_name 即配对键,重复返回 409)
curl -X POST http://127.0.0.1:8000/api/batches/7/screenshots \
  -F "scene_name=01_废弃都市_广场_昼" -F "file=@shot_01.png"

# 3. (可选)自动与同场景+同平台+同画质的最新历史批次对比
curl -X POST http://127.0.0.1:8000/api/batches/7/auto-compare
```

注意:

- **同号覆盖**:上报体 `overwrite=true`(脚本 `report.py --overwrite` 或 manifest `overwrite`;网页手动上报勾选「覆盖同号批次」)时,批次号已存在则**删旧建新**(连带清除旧批次的对比/对比项/基线/热力图);默认 `false` 时同号返回 409,按补传/合并处理。
- 发起对比要求两个批次**场景ID(`scene_id`)相同**(可跨平台);场景不同返回 400。
- `scene_name`(检查点名)是唯一配对键,前后版本必须一致。
- **对比为异步**:`POST /api/comparisons` 返回 `task_id`,前端轮询 `GET /api/comparisons/tasks/{task_id}` 获取进度与结果;同一对批次已对比过直接复用,`force=true` 强制重算。
- `platform` 入库会归一化(WindowsEditor→Windows 等);`p4_version` 可空(界面显示「——」)。

字段与 manifest 映射详见 [docs/上报接入指南.md](上报接入指南.md) 与 [mock_uploads/README.md](../mock_uploads/README.md)。

## 6. API 一览

| 方法 | 路径 | 说明 |
|---|---|---|
| GET | `/api/batches` | 批次列表(场景/画质/P4范围/创建时间/q 过滤,分页) |
| POST | `/api/batches` | 上报批次(scene_id/platform 必填;`overwrite=true` 同号删旧建新) |
| DELETE | `/api/batches/{id}` | 级联删除批次(对比/对比项/基线/图片) |
| DELETE | `/api/batches?created_before=<ISO日期>` | 批量删除该日期之前的批次(级联,created_at < 该日期) |
| POST | `/api/batches/{id}/screenshots` | 上传截图(multipart: scene_name + file) |
| GET | `/api/batches/{id}/screenshots` | 该批次截图列表 |
| POST | `/api/batches/{id}/auto-compare` | 与同场景+同平台+同画质的最新历史批次自动对比 |
| GET | `/api/scenes/{scene_id}/grid` | 列表图矩阵(同场景多批次;支持 `created_dates` 指定多天) |
| GET | `/api/comparisons/lookup` | 只读查一对批次(忽略方向)已有对比 + 各检查点热力图,不触发计算(列表图差异列用) |
| GET | `/api/comparisons` | 对比列表 |
| POST | `/api/comparisons` | 发起对比 `{batch_id, ref_batch_id, force?}`(异步,返回 task_id) |
| GET | `/api/comparisons/tasks/{task_id}` | 轮询对比进度/结果 |
| GET | `/api/comparisons/{id}/scenes` | 检查点结果(status/q/sort/page,含各状态计数) |
| GET | `/api/items/{id}` | 单检查点详情(指标 + 图片 URL + 前后 id) |
| GET | `/api/baselines` | 基线列表 |
| GET | `/api/meta` | 筛选器选项 |

> 对比历史全局上限 100 条,新建超限淘汰最旧。热力图以 WebP 存储(可用 `python -m app.cleanup` 清孤儿文件)。

## 7. 常见问题

**端口被占用** — 后端默认 8000、前端 5173,改端口时同步更新 `frontend/vite.config.js` 的代理目标和 `backend/app/main.py` 的 CORS 白名单。

**想重置演示数据** — `cd backend && .venv\Scripts\python -m app.seed`(会清库并重新生成全部图片),或直接删除 `backend/data/` 目录。

**上传同名检查点报 409** — 同一批次内 `scene_name` 唯一;重传请先删批次或换批次号。

**对比是异步的** — `POST /api/comparisons` 立即返回 `task_id`,后台线程计算、前端轮询进度;
对比记录全局上限 100 条(超限淘汰最旧)。生产化方向:Celery + Redis 任务队列、WebSocket 推送进度。

**磁盘占用** — 截图按上报原样存(通常 JPEG);热力图为 WebP(纯展示派生物)。
清理孤儿图片:`cd backend && .venv\Scripts\python -m app.cleanup`。

**「项目设置」页** — 已可用:配置对比算法阈值(持久化在库)。基线晋升/退役目前仅后端
`promote_baseline` 实现,尚无前端 UI。
