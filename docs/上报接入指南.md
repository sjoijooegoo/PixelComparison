# 数据上报接入指南

本文说明采集端(UE 客户端 / 渲染农场 / CI)如何把一次截图采集的结果上报到
PixelComparison 后端。

## 总体流程

一次上报 = **建批次** + **逐张上传截图**,两步走:

```
1) POST /api/batches                      创建一个批次(批次元信息)
2) POST /api/batches/{id}/screenshots     把该批次的每张截图依次上传
   (对批次内每张图各调一次)
3) POST /api/batches/{id}/auto-compare    (可选)上传完成后自动与历史批次对比
```

采集端通常会先在本地产出一个数据包目录:

```
<批次目录>/
  manifest.json          批次元信息 + 截图清单(见下)
  Screenshot/
    Seq_xxx_0000.png
    Seq_xxx_0001.png
    ...
```

然后由上报脚本读取 `manifest.json`,把字段映射成上面两个接口的请求并发送。
仓库内 `mock_uploads/upload.py` 就是一个**可直接用的参考实现**(仅依赖 Python 标准库)。

- 后端默认地址:`http://127.0.0.1:8000`
- 静态图片访问:上传成功后可经 `/images/batches/{id}/{scene_name}.png` 取回

## 接口一:创建批次

`POST /api/batches`  ·  Content-Type: `application/json`  ·  成功返回 `201`

### 请求体字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | string | 否 | 批次号。省略则后端按时间戳生成。**不能含 `/ \ . ..`** |
| `scene_id` | string | 是 | 场景标识(UE Level / world_name)。**只有同 `scene_id` 的批次能互相对比** |
| `p4_version` | int | 是 | P4 changelist,越大越新(传字符串数字也会被转成整数) |
| `platform` | string | 是 | 平台。会被**归一化**:`WindowsEditor→Windows`、`IOSEditor→iOS`、`AndroidEditor→Android` |
| `creator` | string | 否 | 上报者,默认 `CI机器人` |
| `batch_url` | string | 否 | 真实流水线/构建链接;批次列表里的批次号超链接会用它 |
| `resolution` | string | 否 | 分辨率,如 `1920x1080` |
| `capture_type` | string | 否 | 采集类型,如 `levelsequence` |
| `levelsequence_name` | string | 否 | LevelSequence 名 |
| `levelsequence_path` | string | 否 | LevelSequence 资源路径 |
| `shading_quality` | int | 否 | 画质档位 `0-5`:`5` 电影 / `4` 极致 / `3` 精美 / `2` 均衡 / `1` 流畅 / `0` 节能。缺省时按 `4`(极致)展示 |
| `captured_at` | string | 否 | 采集时间(ISO8601,如 `2024-06-16T10:00:00`);提供则作为批次创建时间,否则用入库时间 |
| `overwrite` | bool | 否 | 默认 `false`。`true` 时若批次号**已存在则删旧建新**:清除旧批次的截图、其参与的对比与对比项、由其晋升的基线、热力图文件,再用本次数据重建。 |

### 行为说明
- 批次号已存在:`overwrite=false`(默认)→ `409`(上报脚本据此"补传截图");`overwrite=true` → 删旧建新并返回 `201`。
- 覆盖为破坏性操作;若该批次正参与"计算中的对比",返回 `409`(稍后再覆盖)。
- `platform` 入库前归一化;`captured_at` 解析失败时静默回退为入库时间。

### 示例
```bash
curl -X POST http://127.0.0.1:8000/api/batches \
  -H "Content-Type: application/json" \
  -d '{
    "id": "2",
    "scene_id": "Village_Dimension_Main",
    "p4_version": 250003,
    "platform": "WindowsEditor",
    "resolution": "1920x1080",
    "capture_type": "levelsequence",
    "levelsequence_name": "Seq_village",
    "batch_url": "https://devops.woa.com/.../executeDetail"
  }'
```

## 接口二:上传截图

`POST /api/batches/{batch_id}/screenshots`  ·  Content-Type: `multipart/form-data`  ·  成功返回 `201`

对批次里的**每一张**截图各调用一次。

### 表单字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `scene_name` | text | 是 | 检查点名,**唯一配对键**:两批对比时按它配对,前后版本必须一致。**不能含 `/ \ . ..`** |
| `file` | file | 是 | PNG 图片 |
| `camera` | text(JSON) | 否 | 相机位姿,如 `{"location":{"x":..,"y":..,"z":..},"rotation":{"pitch":..,"yaw":..,"roll":..}}`,详情页会展示 |
| `frame_index` | int | 否 | 帧序 |

### 行为说明
- 批次不存在 → `404`;同一批次内 `scene_name` 重复 → `409`。
- 图片落盘为 `batches/{batch_id}/{scene_name}.png`。

### 示例
```bash
curl -X POST http://127.0.0.1:8000/api/batches/2/screenshots \
  -F "scene_name=Seq_village_0000" \
  -F "frame_index=0" \
  -F 'camera={"location":{"x":-199174.0,"y":1.2,"z":3.4},"rotation":{"pitch":-10.8,"yaw":68.7,"roll":0.0}}' \
  -F "file=@Screenshot/Seq_village_0000.png;type=image/png"
```

## 接口三:自动对比(可选)

`POST /api/batches/{batch_id}/auto-compare`  ·  无请求体  ·  返回 `202`

把该批次截图**全部上传完成后**调用一次。后端会自动挑一个参照批次发起对比:

- 匹配条件:**同 `scene_id`(场景)+ 同 `platform`(平台)+ 同 `shading_quality`(画质)**,
  且创建时间早于本批次,取其中**最新**的一个。
- 画质为空的旧数据按默认 `4`(极致)等价匹配。
- 找到 → 发起对比(异步,结果可在「对比结果」页查看),返回 `{"matched": true, "ref_batch_id": "...", ...}`。
- 找不到匹配批次 → 返回 `{"matched": false}`,不报错、不创建对比。

> 对比本身要求两批 `scene_id` 相同,故场景必须一致才会匹配。`report.py` 默认在上传完成后
> 自动调用本接口,可用 `--no-auto-compare` 关闭。

### 示例
```bash
curl -X POST http://127.0.0.1:8000/api/batches/2/auto-compare
```

## manifest.json 与字段映射

采集端产出的 `manifest.json` 与接口字段的对应关系(参考实现 `upload.py` 即按此映射):

```json
{
  "format_version": 1,
  "capture_type": "levelsequence",
  "pipeline_data": {
    "id": "2",
    "url": "https://devops.woa.com/.../executeDetail"
  },
  "ue_data": {
    "levelsequence_path": "/Game/.../Seq_village.Seq_village",
    "levelsequence_name": "Seq_village",
    "world_name": "Village_Dimension_Main",
    "platform": "Windows",
    "p4_version": "250003",
    "shading_quality": 5,
    "resolution": { "width": 1920, "height": 1080 }
  },
  "screenshots": [
    {
      "index": 0,
      "name": "Seq_village_0000",
      "image": "Screenshot/Seq_village_0000.png",
      "camera": {
        "location": { "x": -199174.0, "y": 1.2, "z": 3.4 },
        "rotation": { "pitch": -10.8, "yaw": 68.7, "roll": 0.0 }
      }
    }
  ]
}
```

| manifest 路径 | 接口字段 |
|---|---|
| `pipeline_data.id`(兼容旧 `batch_id`) | `POST /api/batches` 的 `id` |
| `pipeline_data.url`(兼容旧 `batch_url`) | `batch_url` |
| `pipeline_data.captured_at`(可缺省) | `captured_at` |
| `pipeline_data.overwrite` 或顶层 `overwrite`(可缺省) | `overwrite`(同号则删旧建新;`report.py --overwrite` 可强制) |
| `ue_data.world_name` | `scene_id` |
| `ue_data.platform` | `platform`(后端归一化) |
| `ue_data.p4_version` | `p4_version`(转 int) |
| `ue_data.resolution{width,height}` | `resolution` 拼成 `"1920x1080"` |
| `ue_data.levelsequence_name/path` | 同名字段 |
| `ue_data.shading_quality`(可缺省) | `shading_quality`(画质档位 0-5) |
| `capture_type` | `capture_type` |
| `screenshots[].name` | 每张图的 `scene_name` |
| `screenshots[].image` | 读取该相对路径的图片作为 `file` |
| `screenshots[].index` | `frame_index` |
| `screenshots[].camera` | `camera`(序列化成 JSON 字符串) |

> 兼容性:`pipeline_data` 同时支持新命名(`id`/`url`)和旧命名(`batch_id`/`batch_url`)。
>
> **`pipeline_data` 整体可省略**:不提供时,批次号由后端自动生成(时间戳),`batch_url` 留空,
> 创建时间取上报当时的时间。即只要有 `ue_data` + `screenshots` 也能上报。

## 用参考脚本上报

### 推荐:`report.py`(指哪传哪,仅依赖标准库)

直接把 `manifest.json` 路径(或其所在目录)丢给它即可。脚本可整文件拷到采集端用。

```powershell
# 传一个数据包(传 manifest.json 或其所在目录均可)
python report.py "C:\path\to\PixelComparison\manifest.json"
python report.py "C:\path\to\PixelComparison"

# 指定后端 IP / 端口(几种写法等价)
python report.py ./pkg --host 10.30.129.32 --port 8000
python report.py ./pkg --base http://10.30.129.32:8000
python report.py ./pkg --base 10.30.129.32:8000          # 省略 http:// 也可
BASE=http://host:8000 python report.py ./pkg             # 也支持 BASE/HOST/PORT 环境变量
```

图片按 `manifest.json` 所在目录解析相对路径;批次/截图已存在(`409`)时自动当作已传、继续补齐;
连不上后端或缺图会给出明确提示并以非零退出码结束(便于 CI 判断成败)。

### 批量:`mock_uploads/upload.py`

用于一次性上报 `mock_uploads` 下所有自带的示例数据包。

```powershell
python mock_uploads/upload.py            # 上报全部示例包
python mock_uploads/upload.py 7          # 仅上报指定包(目录名)
BASE=http://host:8000 python mock_uploads/upload.py
```

两者上报逻辑一致:读 `manifest.json` → `POST /api/batches` → 逐张
`POST /api/batches/{id}/screenshots`(带 `camera`/`frame_index`);批次已存在时继续补传截图。

## 上报后

- 在「批次管理」页可看到新批次(若是经脚本在后台上报的,点页面「刷新」即可更新列表与筛选项)。
- 选两个**同 `scene_id`** 的批次(基线 / 对比)发起对比;`scene_id` 不同会被拦截。
- 对比是另一条链路(`POST /api/comparisons`,异步 + 进度轮询),不属于上报范畴,这里不展开。

## 注意事项

- `id`(批次号)与 `scene_name` 会落地为文件路径段,**禁止包含 `/`、`\`、`.`、`..`**,否则返回 `400`。
- 同一批次内 `scene_name` 必须唯一;同一 `scene_name` 在不同版本批次间要保持一致才能正确配对。
- 接口非幂等:批次重复建 `409`、截图重复传 `409`;补传时忽略 `409` 继续即可。
